# Domain Test Strategy Design

**목표**

- `core/core-api` 기준으로 도메인/유스케이스와 조회 저장소 테스트의 1차 표준을 정의한다.
- 기존 또는 과거 테스트 코드에 의존하지 않고 현재 프로덕션 코드 구조만 기준으로 테스트 전략을 수립한다.
- 결함 조기 발견, 회귀 방어, 향후 테스트 확장 시 일관된 토대를 동시에 확보한다.

**적용 범위**

- 포함: `UseCase`, `ApplicationService`, `DomainService`, `QueryRepository`
- 제외: `Controller`, 외부 연동 통합 테스트, 전수 CRUD 저장소 테스트

## 1. 아키텍처 원칙

테스트는 명령 계열과 조회 계열을 분리해서 운영한다.

- 명령/상태 변경 계열
  - 대상: `UseCase`, `ApplicationService`, `DomainService`
  - 목적: 입력 검증, 사전 조건, 상태 변경 결과, 협력자 호출 순서 검증
  - 방식: 단위 테스트
- 조회/검색 계열
  - 대상: `QueryRepository`
  - 목적: 필터, 정렬, 커서, 조인 기반 DTO 매핑 검증
  - 방식: JPA 슬라이스 테스트

일반 `JpaRepository` 는 Spring Data 기본 CRUD 동작을 검증하지 않는다. 도메인 의미가 있는 커스텀 조회 또는 상태 판별 조건이 있을 때만 별도 테스트한다.

## 2. 테스트 책임 경계

### 명령 계열

명령 테스트는 “이 진입점이 어떤 규칙을 보장하는가”에 집중한다.

- 정상 흐름에서 올바른 결과를 반환하는가
- 잘못된 입력 또는 상태에서 정확한 예외를 던지는가
- 후속 협력자 호출이 비즈니스 순서대로 일어나는가
- 불필요한 DB/스프링 컨텍스트 없이 순수 자바 수준에서 빠르게 검증되는가

명령 테스트에서 저장소, 이벤트 발행기, 외부 매니저는 테스트 더블을 사용한다. 단, 엔티티/값 객체 자체 규칙은 가능한 실제 객체로 검증한다.

### 조회 계열

조회 테스트는 “이 쿼리가 실제 데이터에서 어떤 결과를 반환하는가”에 집중한다.

- 검색 필터가 정확히 적용되는가
- 정렬 기준과 tie-breaker 가 일관되게 적용되는가
- 커서 기반 페이지네이션이 누락/중복 없이 동작하는가
- 조인 결과가 응답 DTO 에 정확히 투영되는가

조회 테스트는 `@DataJpaTest` 와 H2 기반으로 작성하고, 최소 데이터셋을 명시적으로 구성한다. Querydsl 쿼리 자체를 mock 으로 검증하지 않는다.

## 3. 우선순위

### 1순위

- `order`
- `hold`
- `performanceseat`

사유: 좌석 선점, 주문 상태, 중복 요청, 제한 수량 등 실패 비용이 높은 규칙이 집중될 가능성이 크다.

### 2순위

- `show`
- `performance`
- `showlike`

사유: 검색/정렬/커서/상세 조합 조회 로직과 사용자별 상태 조회가 많다.

### 3순위

- `auth`
- `member`
- `commoncode`

사유: 외부 의존성 비율이 높거나 상대적으로 규칙 복잡도가 낮다.

## 4. 테스트 작성 표준

### 네이밍

- 클래스: `대상클래스명Test`
- 메서드: `조건_행위_결과` 또는 `상황_예외` 패턴

예시:

- `중복된_좌석_ID가_있으면_예외를_던진다`
- `보류중인_주문이_이미_존재하면_주문을_시작할_수_없다`
- `카테고리와_지역으로_필터링된_공연_목록을_조회한다`

### 구조

- 한 테스트는 하나의 규칙만 검증한다.
- 테스트 내부에 과도한 분기나 반복을 넣지 않는다.
- assertion 은 결과 의미가 드러나도록 구체적으로 작성한다.

### fixture

- 도메인 의미가 있는 테스트 데이터 빌더를 직접 작성한다.
- Fixture Monkey 는 대량 데이터 생성 보조 용도로만 제한적으로 사용한다.
- 랜덤성에 의존하지 않는다.

### mock 사용 기준

- 협력자 경계에서는 허용한다.
- 쿼리 결과나 도메인 규칙을 mock behavior 로 대체하지 않는다.
- mock verification 은 호출 자체보다 비즈니스 효과가 불충분할 때만 사용한다.

## 5. 도메인별 전략

### `StartOrderUseCase` 축

아래 항목을 우선 검증한다.

- 좌석 ID 중복
- 빈 좌석 목록
- 공연별 최대 선점 수량 초과
- 같은 회원/공연의 `PENDING` 주문 중복
- 정상 주문 시작 시 hold 생성, 이벤트 발행, 주문 생성, 이력 기록

### `SelectSeatUseCase` 축

- 선택 가능 여부 검증 실패 시 후속 처리 없음
- 정상 선택 시 선택 처리 후 상태 이벤트 발행
- 동일 좌석 잠금 경계에서 핵심 규칙이 유지되는지 단위 수준에서 보장

### `ShowListQueryRepository` 축

- 최신순/인기순/오픈임박순 정렬
- 카테고리, 지역, 장르, 키워드 필터
- 커서 기반 다음 페이지 계산
- 응답 DTO 의 venue/genre 매핑

## 6. 초기 구현 범위

1차 구현은 다음 두 축으로 시작한다.

1. `StartOrderUseCase` 단위 테스트
2. `ShowListQueryRepository` 슬라이스 테스트

선정 이유:

- 명령/조회 두 축의 표준을 동시에 정립할 수 있다.
- 이후 `performanceseat`, `show detail`, `order detail` 로 확장할 때 재사용 가능한 테스트 패턴을 얻을 수 있다.

## 7. 검증 전략

- 단위 테스트는 대상 클래스 단위로 실행해 red-green 을 확인한다.
- 슬라이스 테스트는 저장소 클래스 단위로 실행한다.
- 마지막에는 최소한 `:core:core-api:test --tests ...` 수준으로 관련 테스트 묶음을 다시 실행한다.

## 8. 남은 리스크

- 분산락 AOP 자체는 이번 범위에서 직접 검증하지 않는다.
- Oracle 전용 SQL 차이는 H2 기반 슬라이스 테스트에서 완전히 대체되지 않을 수 있다.
- 조회 테스트가 증가하면 공통 fixture 와 데이터셋 조립 유틸이 필요해질 수 있다.
