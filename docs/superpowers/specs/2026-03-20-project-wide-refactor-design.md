# Project-Wide Refactor Design

## 목표

기존 공개 API 계약과 외부 동작을 유지하면서, 프로젝트 전반에 다음 규약을 순차 적용한다.

- TDD 기반 리팩토링
- OOP 책임 분리
- 들여쓰기 depth 2단계 유지
- 메서드 15라인 이하 지향
- 원시값 포장과 일급 컬렉션 적용
- `else` 지양, Early Return 우선

## 범위

대상은 `core-api` 전반이며, 도메인별로 순차 진행한다.

1. `order/hold`
2. `performanceseat`
3. `queue`
4. `show`
5. `auth/member`

## 비범위

- 공개 API 경로 변경
- 요청/응답 JSON 필드 변경
- 상태코드, 헤더, 외부 식별자 규약 변경
- DB 스키마 변경
- Redis key 규약 변경
- 요청하지 않은 기능 추가

## 설계 원칙

### 1. 계약 우선

각 도메인 시작 전에 공개 API 계약 테스트를 먼저 고정한다.

- 컨트롤러 경로
- 상태코드
- 응답 헤더
- 응답 JSON 구조
- 대표 검증 오류 응답

### 2. 내부 값 객체 강화

리팩토링 대상 도메인마다 아래 후보를 우선 검토한다.

- 식별자
- 검색 조건
- 상태 전이 입력
- 좌석/토큰/정렬 조건 컬렉션

단일 primitive를 바로 여러 메서드에 넘기는 구조를 줄이고, 검증과 정규화를 값 객체 내부로 이동한다.

### 3. 유스케이스는 orchestration만 담당

유스케이스는 다음 책임만 남긴다.

- 인증 사용자 확인
- 조회/검증 흐름 조합
- 도메인 서비스 호출
- 이벤트 발행

입력 정규화, 중복 검증, 상태 규칙, 컬렉션 가공은 별도 타입으로 이동한다.

### 4. 작은 커밋

도메인별로 아래 단위로 끊는다.

1. 계약 테스트
2. 값 객체 또는 일급 컬렉션
3. 긴 메서드 분리
4. 회귀 테스트
5. 커밋

## 도메인별 방향

### order/hold

- HOLD 생성 계약 고정
- 주문 시작 입력 컬렉션 값 객체화
- 주문 시작/종료 유스케이스를 orchestration 중심으로 단순화

### performanceseat

- 선택/해제/상태조회 계약 고정
- 좌석 선택 컬렉션과 상태 계산 로직을 값 객체 또는 정책 타입으로 이동
- 좌석 관련 command/query usecase의 긴 메서드 분리

### queue

- 진입/상태/이탈 계약 고정
- queue entry id, token, admission policy 입력을 값 객체로 검토
- 정책 해석과 runtime store 접근 책임 분리

### show

- 목록/상세/검색 계약 고정
- 정렬 조건, 커서, 검색 조건 컬렉션 값 객체화
- query helper와 usecase 책임 경계 정리

### auth/member

- 로그인/토큰/회원 조회 계약 고정
- 이메일, 토큰, principal 관련 primitive 전달 축소
- 인증/회원 usecase 메서드 분해

## 검증 전략

- 새 타입 추가 시 전용 단위 테스트 작성
- 리팩토링 대상 유스케이스 기존 테스트 유지
- 도메인 완료 전 계약 테스트와 핵심 유스케이스 테스트 묶음 실행
- 커밋 전 검증 명령 실행 결과 확인

## 커밋 전략

커밋 메시지는 기존 규약을 따른다.

- `refactor(order): ...`
- `refactor(performanceseat): ...`
- `refactor(queue): ...`
- `refactor(show): ...`
- `refactor(auth): ...`

각 커밋은 한 도메인 또는 한 하위 책임만 다룬다.
