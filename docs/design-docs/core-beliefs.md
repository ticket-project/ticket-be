# Core Beliefs — 설계 원칙

## 1. 모듈 경계가 곧 규율이다

core-api와 core-domain의 경계는 "권장"이 아니라 "강제"다. ArchUnit이 빌드 시점에 위반을 잡아낸다. 경계가 모호한 코드는 곧 에이전트가 재생산하는 나쁜 패턴이 된다.

## 2. 인프라는 도메인을 모른다

Redis, 메시징, HTTP 클라이언트는 `..infra..` 패키지 뒤에 숨긴다. 도메인 로직은 인터페이스에만 의존하고, 구현체는 인프라가 제공한다. 이를 통해 테스트에서 인프라를 쉽게 대체할 수 있다.

## 3. 에이전트가 읽을 수 없으면 존재하지 않는 것이다

Slack, Google Docs, 사람 머릿속에만 있는 지식은 시스템에 반영되지 않는다. 모든 설계 결정, 제약 조건, 도메인 규칙은 리포지터리 내 마크다운 또는 코드로 기록한다.

## 4. 문서화보다 코드로 강제한다

"이렇게 하지 마세요"라는 문서보다 린터/테스트가 자동으로 잡아내는 것이 낫다. 문서로 충분하지 않으면 규칙을 코드(ArchUnit, 커스텀 린터, CI 검증)로 승격한다.

## 5. TTL이 있으면 만료 후 처리도 있다

Redis TTL을 설정하면 만료 후의 상태 정합성 처리가 반드시 동반되어야 한다. "Redis가 알아서 삭제해준다"는 DB 측 정합성을 보장하지 않는다.

## 6. 작은 변경을 자주 병합한다

대규모 PR보다 작고 자체 완결적인 PR을 선호한다. 에이전트가 하루에 여러 PR을 생성하는 흐름에서, 작은 PR은 리뷰 비용이 낮고 롤백이 쉽다.

## 7. 경계에서 파싱한다

외부 입력(API 요청, Redis 데이터, DB 결과)은 시스템 경계에서 즉시 파싱하고 검증한다. 경계를 넘어온 데이터는 항상 유효하다고 가정할 수 있어야 한다.

## 8. 테스트가 아키텍처를 지킨다

ArchUnit 테스트는 "있으면 좋은 것"이 아니라 아키텍처 무결성의 마지막 방어선이다. 새로운 아키텍처 규칙이 생기면 반드시 테스트로 인코딩한다.
