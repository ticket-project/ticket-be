# Ticket Backend Product Overview

## 1. 이 프로젝트가 만드는 것

이 프로젝트는 공연/전시 예매 서비스의 백엔드다. 사용자는 쇼를 조회하고, 회차별 좌석 상태를 확인하고, 좌석을 선택하거나 선점한 뒤 주문을 만들 수 있다.

현재 구현은 "예매 직전까지의 핵심 흐름"에 집중되어 있다.

- 로그인/인증
- 쇼/회차/좌석 조회
- 좌석 선택(selection)
- 좌석 선점(hold)
- 주문 생성, 조회, 취소, 만료

결제와 대기열은 다음 단계로 계획 중이다.

## 2. 현재 사용자 흐름

### 2.1 로그인

사용자는 이메일 로그인 또는 소셜 로그인 연동 흐름을 통해 인증할 수 있다.

### 2.2 쇼 탐색

사용자는 다음 정보를 조회할 수 있다.

- 쇼 목록
- 쇼 상세
- 최신 등록 쇼
- 예매 오픈 예정 쇼
- 장르/카테고리 기반 검색
- 회차별 좌석 정보
- 공연장 레이아웃

### 2.3 좌석 선택

좌석 선택은 UX 보조 상태다.

- 사용자가 좌석을 클릭하면 Redis에 임시 선택 상태가 저장된다.
- 선택은 자동 만료된다.
- 선택 상태는 실시간으로 다른 사용자에게 전파된다.

중요:

- 선택했다고 해서 좌석이 보장되지는 않는다.
- 다른 사용자가 더 먼저 hold를 성공하면 selection보다 hold가 우선한다.

### 2.4 좌석 선점

실제 예매 권리는 hold에서 확보된다.

- 사용자는 좌석 목록을 선점 요청한다.
- 서버는 회차 유효성, 좌석 상태, 수량 제한을 검증한다.
- 선점에 성공하면 일정 시간 동안 해당 좌석은 다른 사용자가 사용할 수 없다.
- 동시에 `PENDING` 주문이 생성된다.

현재 정책:

- 같은 회원은 같은 회차에 `PENDING` 주문 1건만 가능
- hold는 여러 좌석을 한 번에 묶어서 처리
- hold는 만료 시 자동 해제

### 2.5 주문

hold가 성공하면 주문이 바로 만들어진다.

- 외부 식별자는 `orderKey`
- 주문은 기본적으로 `PENDING`
- 사용자는 주문 상세를 조회할 수 있다
- 사용자는 `PENDING` 주문을 취소할 수 있다
- 만료된 주문은 자동 종료된다

## 3. 현재 구현 범위와 미구현 범위

### 구현 완료

- 인증/회원
- 쇼/회차/좌석 조회
- 실시간 좌석 선택
- Redis 기반 좌석 선점
- `PENDING` 주문 생성 및 종료
- Redis 만료 이벤트 기반 즉시 후처리

### 계획 중

- 결제 시작/성공/실패/콜백
- 좌석 최종 판매 확정
- 대기열 토큰 발급 및 입장 제어
- 운영자용 지표/모니터링 보강

## 4. 이 서비스의 핵심 차별점

현재 코드 기준으로 이 서비스가 해결하는 문제는 아래와 같다.

- 여러 사용자가 동시에 같은 좌석을 잡는 경쟁 상황 제어
- selection, hold, order를 분리해서 사용자 경험과 실제 점유를 구분
- Redis TTL과 listener를 통해 만료 후처리를 빠르게 반영
- WebSocket으로 좌석 상태를 실시간 전파

## 5. 앞으로 붙을 기능

### 결제

예정 목표:

- hold로 확보한 좌석을 결제 성공 시 최종 확정
- 결제 실패/이탈 시 hold 해제
- 주문 상태를 `CONFIRMED`, `PAYMENT_FAILED` 까지 완성

### 대기열

예정 목표:

- 인기 공연/회차에 대한 동시 진입 제어
- 대기 번호, 허용 상태, TTL 기반 입장 관리
- hold/order 이전 단계에서 과도한 트래픽을 흡수

- 대기열이 없으므로 초고트래픽 상황은 아직 완성되지 않았다
